Every raw finding was then handed to **two independent skeptics told to _refute_ it** against the real code — one on correctness, one on reachability. **Severity note:** the `[sev]` shown is the finder's original; `→` shows the skeptics' recalibrated severity where they moved it. The skeptics were consistently more conservative (e.g. both P0s downgraded on blast-radius).

## 1. [P0] → **P2** (verifiers) First-touch founder projection is non-atomic: partial failure orphans a person entity and permanently strands the company
`api/src/memory/resolve.js:44` · memory-atom · class: `partial-write-corruption` · ★ double-confirmed

**Failure scenario.** journey.js:18 calls resolveOrProjectFounder(request.authUser) with NO client, so client defaults to the pool (db.js:7) — every statement autocommits, there is no BEGIN/COMMIT anywhere in src/memory. migrateProfile -> createFounderAndCompany (nodes.js:29-38) inserts the PERSON first (committed, ref=sub), THEN the company, THEN the founded edge, THEN a loop of assertClaim inserts. If any statement after the person INSERT fails (connection drop, pool timeout, CHECK/constraint error, process restart), the person row with ref=sub stays committed while company/edge/claims are missing. On the next request byRef (resolve.js:33) hits the fast path and returns the orphan person, so migrateProfile NEVER re-runs; companyForFounder (journey.js:21) finds no founded edge and returns company:null forever. The founder's kernel company_name is silently lost and the state cannot self-heal.

**Evidence.** resolve.js:29 `resolveOrProjectFounder(authUser, client = pool)`; journey.js:18 passes no client; nodes.js:29-38 does 3 sequential inserts + migrate.js:20-25 loops assertClaim, all on the same non-transactional `pool`; grep for BEGIN/COMMIT/connect in src/memory returns nothing.

**Fix.** Acquire one client via pool.connect(), BEGIN before byRef-miss work, run createFounderAndCompany + all assertClaim on that client, COMMIT on success, ROLLBACK on any error, release in finally. Pass that client all the way down.

## 2. [P0] → **P1** (verifiers) Unauthenticated SSRF / internal port scan: user-supplied domain drives recon with no private-IP validation
`api/src/services/recon-ports.js:52` · coldread-pipeline · class: `injection` · ★ double-confirmed

**Failure scenario.** POST /api/session/start {"website_url":"http://169.254.169.254"} (or "http://10.0.0.5", "localhost:6379") on the public no-login endpoint. extractDomain() (recon-pipeline.js:196) yields hostname `169.254.169.254`/`localhost`, then reconPorts TCP-connects to that host on 13 ports incl. 3306/5432/6379/9200/27017, reconHttp (recon-http.js:87) issues https.request to it with rejectUnauthorized:false, and reconIp (recon-ip.js:34) resolves/enriches it. Nothing anywhere checks that the target resolves to a public/global IP. proof360's EC2 box becomes an internet-facing scanner of its own VPC / cloud metadata / internal services, and open-port results are reflected back to the caller as report findings.

**Evidence.** recon-ports.js:44-45 `socket.connect(port, host)` with host = raw domain; recon-pipeline.js:196-203 extractDomain does no IP-class validation; session-start.js:13-18 only checks presence of website_url. No allowlist / no isPrivate/isLoopback/isLinkLocal guard exists in the pipeline.

**Fix.** Before any recon source runs, resolve the hostname and reject if it (or any resolved A/AAAA) is loopback, private (10/8, 172.16/12, 192.168/16), link-local (169.254/16, incl. cloud metadata), CGNAT (100.64/10), or non-global; also reject bare-IP inputs and re-validate on every redirect. Fail the whole pipeline closed for such hosts.

## 3. [P1] → **P2/P1** (verifiers) DNS lookup failure is silently rendered as 'missing' DMARC/SPF, firing false critical/high gaps and corrupting trust_score
`api/src/services/recon-dns.js:87` · coldread-pipeline · class: `silent-failure` · ★ double-confirmed

**Failure scenario.** Any transient resolver failure (timeout, SERVFAIL, egress hiccup) on the _dmarc TXT or root TXT lookup makes lookupTxt() catch and return []. parseDmarcPolicy([]) -> 'missing' and parseSpfPolicy([]) -> 'missing'. reconDns then returns a fully-formed object (not an error object), so extractReconContext sets ctx.dmarc_policy='missing' / ctx.spf_policy='missing', which are exactly the values the dmarc gap (severity high, -10) and spf gap (medium, -5) trigger on. The report shows 'Email domain protection gap (DMARC)' with evidence citation 'Passive DNS lookup — live record at time of assessment' even though DNS was never successfully read. A company with correct p=reject DMARC gets penalized and shown a false failing control.

**Evidence.** recon-dns.js:16-19 lookupTxt catch returns []; :42 parseDmarcPolicy returns 'missing' for empty records; :50-53 parseSpfPolicy returns 'missing' for empty; :83 anyFailed only true if a promise *rejects*, but the helpers never reject; extractReconContext recon-pipeline.js:89 `if (d.dmarc_policy) ctx.dmarc_policy = d.dmarc_policy` accepts 'missing'; gaps.js:212 dmarc trigger `['missing','none'].includes(ctx.dmarc_policy)`.

**Fix.** Distinguish 'lookup failed' from 'record absent'. Have lookupTxt/lookupMx re-throw or return a sentinel on real DNS errors (vs NXDOMAIN/empty), set dmarc_policy/spf_policy to null (unknown) on failure, and make gap triggers require an affirmatively-observed 'missing' rather than treating unknown as missing.

## 4. [P1] → **P2** (verifiers) Health-check curl races pm2 restart; connection-refused (exit 7) aborts the deploy under set -e (false red, both paths)
`scripts/deploy.sh:136` · config-deploy · class: `silent-failure` · ★ double-confirmed

**Failure scenario.** After the pm2 restart the script sleeps only 3s (deploy.sh:135) then runs STATUS=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:${PORT}/api/health 2>/dev/null) (deploy.sh:136). If the node process has not yet bound the port, curl exits 7. Because the script runs under set -e (deploy.sh:2), the variable assignment inherits curl's non-zero status and the shell aborts immediately — before the `if [ "$STATUS" = "200" ]` check at deploy.sh:137, so the intended WARNING branch is never reached and the deploy reports failure for a healthy-but-slow API. Verified locally: `set -e; S=$(curl ...refused)` exits 7 without executing the following line. Identical bug in the CI remote script (deploy.yml:164 under set -e at deploy.yml:75): the abort marks the SSM command Failed, fails the 'Wait for deploy' step, and skips the Telegram webhook registration at deploy.yml:167-168.

**Evidence.** deploy.sh:2 `set -e`; deploy.sh:135-136 `sleep 3` then `STATUS=$(curl ... 2>/dev/null)`; local repro printed only `exit=7` and never the post-curl echo.

**Fix.** Make the health probe non-fatal and give it a bind window: `STATUS=$(curl ... || true)` (or `set +e` around it) and poll in a retry loop (e.g. 10 attempts x 2s) before deciding, so a slow bind does not fail the deploy.

## 6. [P2] → **P2** (verifiers) Concurrent first-touch mints duplicate file-kernel profiles and loses the loser's transactions
`api/src/services/memory-store-file.js:181` · memory-atom · class: `race` · ★ double-confirmed

**Failure scenario.** getOrCreateFounder (148-179) and getOrCreateActiveProfile (181-204) use existsSync-then-writeJsonAtomic with NO lock, unlike appendTransaction which is withLock-guarded. Two concurrent first authenticated touches for the same founder both see active-profile.json absent, each generate a DIFFERENT profileId, each ensureProfileDirs + writeJsonAtomic(active-profile.json) — last writer wins. Result: two profile directories exist; active-profile.json points at only one; the other is orphaned. **Evidence.** memory-store-file.js:184 `if (existsSync(path)) return readJson(path);` then 188 `const profileId = randomUUID();` with no surrounding withLock; contrast appendTransaction:259 which does wrap its critical section in withLock.

**Fix.** Wrap founder + active-profile creation in withLock keyed on the founder hash so the check-and-create is atomic.

## 7. [P2] → **P2** (verifiers) Stale lock directory permanently wedges a profile's append path
`api/src/services/memory-store-file.js:100` · memory-atom · class: `silent-failure` · ★ double-confirmed

**Failure scenario.** withLock acquires by mkdir(lockPath) and only removes it in the finally of a successful run. If the process crashes or is killed while holding the lock (between mkdir at 106 and rm at 124), the lock directory survives. There is no liveness/TTL reclaim — owner.json records a pid (108) but nothing ever reads it — so every subsequent appendTransaction/writeSnapshot/claimSession spins for LOCK_TIMEOUT_MS and then throws memory_lock_timeout forever. That founder's memory becomes permanently unwritable until a human deletes the dir.

**Evidence.** memory-store-file.js:104-119 acquire loop only handles EEXIST by sleeping until timeout; 114 `throw new Error('memory_lock_timeout')`; owner.json written at 107 is never read back; rm only in finally at 124.

**Fix.** On EEXIST, read owner.json; if its pid is dead (process.kill(pid,0) throws ESRCH) or created_at older than a TTL, reclaim the lock (rm + retry) instead of waiting out the timeout.

## 10. [P2] Awaited-URL reply is silently dropped (and the wait never clears) when a cold-read session already exists
`frontend/src/pages/Chat.jsx:1731` · cer-state · class: `silent-failure` · ⚠ single-verdict (under-verified — spend limit)

**Failure scenario.** Founder ran a cold-read earlier that returned no company name (analysis.company_name null), so companyData.session_id is set but companyProfile.name is still null. ## 11. [P2] OAuth `state` is a static constant, not a per-request CSRF nonce (login CSRF / session fixation)
`frontend/src/pages/Chat.jsx:212` · frontend-injection · class: `client-trust` · ⚠ single-verdict (under-verified — spend limit)

**Failure scenario.** loginGoogle sends state:'google' (implicit `response_type: token`) and loginAuth0 sends state:'auth0' (line 230). The callback only checks the value against those constants (Portal.jsx:107 `searchParams.get('state') === 'auth0'`; line 117 forwards `params.get('state')` from the hash for the implicit flow) — it is used purely to route which handler runs, never as an unguessable value bound to the initiating browser. Attacker logs into their own Google account, captures the returned access_token, then lures a victim to `https://proof360.au/portal/callback#access_token=<ATTACKER_TOKEN>&state=google`. The app accepts it because state matches the constant, writes `founder_auth`/`portal_auth`, and the victim is now silently operating inside the attacker's identity — any reports/CERs the victim saves land in the attacker-controlled account (login CSRF / session fixation).

**Evidence.** Chat.jsx:212 `state: 'google'` and Chat.jsx:230 `state: 'auth0'` are hardcoded strings; Portal.jsx:106-117 compares state only to those constants and never generates/stores/verifies a random nonce (no crypto.getRandomValues for state, no sessionStorage state round-trip).

**Fix.** Generate a random state (crypto.getRandomValues), persist it in sessionStorage before redirect, and reject the callback unless the returned state exactly matches the stored value. Keep the flow-routing tag (google/auth0/microsoft) as a separate field or encode it alongside the nonce, but validate the nonce independently.

## 12. [P2] Shareable cold-read URL uses unsigned, forgeable params and targets a deleted route
`frontend/src/utils/shareable-url.js:11` · frontend-injection · class: `forgeable-token` · ⚠ single-verdict (under-verified — spend limit)

**Failure scenario.** buildShareableUrl emits `/audit/cold-read?session=<id>&url=<encoded>` with the session id and url as plaintext, integrity-free query params. ## 16. [P3] CER event fold orders by wall-clock `ts` instead of append sequence, so a non-monotonic clock yields the wrong current status
`api/src/services/cer-projection.js:121` · cer-state · class: `race` · ⚠ single-verdict (under-verified — spend limit)

**Failure scenario.** reconstruct() pushes cer_events into the snapshot in true append (transaction) order, but eventsForCer() throws that reliable order away and re-sorts by `String(a.ts).localeCompare(String(b.ts))`. admin_status is then `evs.filter(status-updated).at(-1)?.to` — the last event by ts, not by causal order. Sequence: (1) Submitted; (2) admin -> 'Under review' written at ts=T2; (3) the box's clock steps backward (NTP step, VM time sync, DST-free UTC is safe but container clock resets are not); (4) admin -> 'Booked' — at write time buildStatusUpdatedRecord validates from the live projected admin_status ('Under review')->'Booked', which is legal, and stamps ts=T1 < T2. On read, eventsForCer sorts [Booked@T1, UnderReview@T2]; `.at(-1).to` = 'Under review'. The CER displays 'Under review' even though the founder/admin's latest committed action was 'Booked'. The append log knew the correct order; the ts-sort discarded it.

**Evidence.** Line 121 sorts by ts via localeCompare; lines 131 and 133 both rely on `.at(-1)` over that ts-sorted array to pick current consent and current admin_status. The snapshot's cer_events array (memory-store-file.js reconstruct, line 308) is already in append order and is the authoritative causal order.

**Fix.** Fold in append order — carry a monotonic per-transaction sequence onto each cer_event during reconstruct and sort by that (or don't re-sort at all, since cer_events is already appended in order). Use ts only as a tiebreaker, never as the primary key.

## 18. [P3] Admin status endpoint allows transitions on a consent-withdrawn CER; admin_status audit trail diverges from the founder's withdrawal

## 19. [P3] Extracted URL passed to server scrape pipeline with no scheme/host validation (SSRF entry surface), unauthenticated
`frontend/src/utils/url.js:16` · frontend-injection · class: `injection` · ⚠ single-verdict (under-verified — spend limit)

## 20. [P3] BrowserPanel renders arbitrary user-entered URLs in an iframe with no `sandbox` attribute
`frontend/src/pages/Chat.jsx:674` · frontend-injection · class: `client-trust` · ⚠ single-verdict (under-verified — spend limit)

**Failure scenario.** The mini-browser loads any user-typed URL (BrowserPanel.addTab, line 492) into `<iframe src={activeTab.url}>` (lines 674 and 684) with no `sandbox` attribute, so the framed third-party site runs scripts, opens popups, submits forms and can attempt top-level navigation with full privileges. **Evidence.** Chat.jsx:674-690 both `<iframe>` elements set only `src`/`title`/`style` with no `sandbox`; grep confirms `setPreviewUrl` is invoked exactly once, with `null` (line 1589), so the panel is presently dead code.
